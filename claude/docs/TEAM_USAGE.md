# 团队使用指南（UCC）

本文面向“个人 + 团队”共同使用同一套 Claude Code 配置的场景，目标是：

- 降低多人同时使用时的误用与冲突
- 让配置变更可审查、可回滚、可验证

## 1. 配置标识（Namespace）

本配置集的唯一标识：`UCC`

推荐调用方式：在需要明确使用本配置集的请求中加入 `@ucc`。

判定是否命中本配置集：输出末尾应出现一行 `配置标识：UCC`。

## 2. 推荐工作流

- 复杂需求先 `/ucc-plan`
- 新功能或修复优先 `/ucc-tdd`
- 改完代码必须 `/ucc-code-review`（或语言专用 reviewer）
- 发布前关键路径执行 `/ucc-e2e`（如果项目具备 E2E）

## 3. 团队协作约定

- 配置变更需走代码审查：至少 1 人审查通过
- 变更后必须运行校验：

```bash
node scripts/validate-config.js
node tests/run-all.js
```

- 变更类型建议按范围提交：
  - `docs:` 文档
  - `rules:` 规则
  - `agents:` 代理
  - `commands:` 命令
  - `skills:` 技能

## 4. 常见问题排查

### 4.1 命令/规则看起来没生效

- 在请求里加上 `@ucc`
- 检查输出末尾是否出现 `配置标识：UCC`

### 4.2 团队成员各自改出了不同风格

- 先统一 `rules/common/`（团队必须共识的部分）
- 再在语言目录中做少量差异化（`rules/javascript/`、`rules/typescript/`、`rules/golang/`）

## 5. 不建议做的事

- 不要把密钥写进仓库（即使是示例）
- 不要把团队特定路径硬编码在规则里（例如某个人的本机路径）
