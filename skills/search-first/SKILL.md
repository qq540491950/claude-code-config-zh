# /search-first — 先研究后编码

将"在实现之前搜索现有解决方案"工作流系统化。

## 触发条件

当以下情况使用此技能：
- 开始一个可能有现有解决方案的新功能
- 添加依赖或集成
- 用户要求"添加 X 功能"而你正要写代码
- 在创建新的工具函数、帮助函数或抽象之前

## 工作流程

```
┌─────────────────────────────────────────────┐
│  1. 需求分析                                │
│     定义需要什么功能                         │
│     确定语言/框架约束                        │
├─────────────────────────────────────────────┤
│  2. 并行搜索                                │
│     ┌──────────┐ ┌──────────┐ ┌──────────┐  │
│     │  npm /   │ │  MCP /   │ │  GitHub / │  │
│     │  PyPI    │ │  Skills  │ │  Web      │  │
│     └──────────┘ └──────────┘ └──────────┘  │
├─────────────────────────────────────────────┤
│  3. 评估                                    │
│     评分候选方案（功能、维护性、             │
│     社区、文档、许可证、依赖）               │
├─────────────────────────────────────────────┤
│  4. 决策                                    │
│     ┌─────────┐  ┌──────────┐  ┌─────────┐  │
│     │  直接   │  │  扩展    │  │  自建   │  │
│     │  采用   │  │  /封装   │  │  定制   │  │
│     └─────────┘  └──────────┘  └─────────┘  │
├─────────────────────────────────────────────┤
│  5. 实现                                    │
│     安装包 / 配置 MCP /                     │
│     编写最小化自定义代码                     │
└─────────────────────────────────────────────┘
```

## 决策矩阵

| 信号 | 行动 |
|------|------|
| 完全匹配，维护良好，MIT/Apache 许可 | **直接采用** — 安装并直接使用 |
| 部分匹配，基础良好 | **扩展** — 安装 + 编写薄封装 |
| 多个弱匹配 | **组合** — 结合 2-3 个小包 |
| 没有合适的 | **自建** — 编写定制代码，但基于研究 |

## 如何使用

### 快速模式（内联）

在编写工具函数或添加功能之前，心里过一遍：

0. 仓库中是否已存在？→ 先用 `rg` 搜索相关模块/测试
1. 这是常见问题吗？→ 搜索 npm/PyPI
2. 是否有 MCP 支持？→ 检查 `~/.claude/settings.json` 并搜索
3. 是否有技能支持？→ 检查 `~/.claude/skills/`
4. 是否有 GitHub 实现/模板？→ 运行 GitHub 代码搜索

### 完整模式（代理）

对于非简单功能，启动研究代理：

```
Task(subagent_type="general-purpose", prompt="
  研究现有工具用于：[描述]
  语言/框架：[语言]
  约束：[任何约束]

  搜索：npm/PyPI、MCP 服务器、Claude Code 技能、GitHub
  返回：带建议的结构化比较
")
```

## 按类别搜索快捷方式

### 开发工具
- 代码检查 → `eslint`、`ruff`、`textlint`
- 代码格式化 → `prettier`、`black`、`gofmt`
- 测试 → `jest`、`vitest`、`pytest`、`go test`
- 提交前检查 → `husky`、`lint-staged`、`pre-commit`

### AI/LLM 集成
- Claude SDK → Context7 获取最新文档
- 提示词管理 → 检查 MCP 服务器
- 文档处理 → `unstructured`、`pdfplumber`、`mammoth`

### 数据与 API
- HTTP 客户端 → `ky`/`got` (Node)、`httpx` (Python)
- 数据验证 → `zod` (TS)、`pydantic` (Python)
- 数据库 → 先检查 MCP 服务器

### 内容与发布
- Markdown 处理 → `remark`、`unified`、`markdown-it`
- 图片优化 → `sharp`、`imagemin`

## 与其他代理集成

### 与 planner 代理
规划器应在第 1 阶段（架构审查）之前调用研究器：
- 研究器识别可用工具
- 规划器将其纳入实现计划
- 避免在计划中"重复造轮子"

### 与 iterative-retrieval 技能
结合用于渐进式发现：
- 循环 1：广泛搜索（npm、PyPI、MCP）
- 循环 2：详细评估热门候选
- 循环 3：测试与项目约束的兼容性

## 示例

### 示例 1："添加死链检查"
```
需求：检查 markdown 文件中的断链
搜索：npm "markdown dead link checker"
发现：textlint-rule-no-dead-link（评分：9/10）
行动：直接采用 — npm install textlint-rule-no-dead-link
结果：零自定义代码，久经考验的解决方案
```

### 示例 2："添加 HTTP 客户端封装"
```
需求：带重试和超时处理的弹性 HTTP 客户端
搜索：npm "http client retry"
发现：got（Node）带重试插件
行动：直接采用 — 使用 got 并配置重试
结果：零自定义代码，生产级库
```

### 示例 3："添加配置文件检查器"
```
需求：根据 schema 验证项目配置文件
搜索：npm "config linter schema"
发现：ajv-cli（评分：8/10）
行动：采用 + 扩展 — 安装 ajv-cli，编写项目特定 schema
结果：1 个包 + 1 个 schema 文件，无自定义验证逻辑
```

## 反模式

- **急于编码**：不检查是否存在就写工具函数
- **忽略 MCP**：不检查 MCP 服务器是否已提供该能力
- **过度定制**：过度封装库导致失去其优势
- **依赖膨胀**：为一个小功能安装巨大的包
