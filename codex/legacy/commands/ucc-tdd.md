# TDD 命令

此命令调用 **tdd-guide** 代理，强制执行测试驱动开发方法论。

## 命令功能

1. **搭建接口** - 先定义类型/接口
2. **先生成测试** - 编写失败的测试（红）
3. **实现最小代码** - 只写足够通过的代码（绿）
4. **重构** - 改进代码同时保持测试通过（重构）
5. **验证覆盖率** - 确保 80%+ 测试覆盖率

## 何时使用

以下情况使用 `/ucc-tdd`：
- 实现新功能
- 添加新函数/组件
- 修复 Bug（先写重现 Bug 的测试）
- 重构现有代码
- 构建关键业务逻辑

## 工作原理

tdd-guide 代理会：

1. **定义接口** 输入/输出
2. **编写将失败的测试**（因为代码还不存在）
3. **运行测试** 验证它们因正确原因失败
4. **编写最小实现** 让测试通过
5. **运行测试** 验证通过
6. **重构** 代码同时保持测试绿色
7. **检查覆盖率** 如低于 80% 添加更多测试

## TDD 循环

```
红 → 绿 → 重构 → 重复

红：      编写失败的测试
绿：    编写最小代码使其通过
重构： 改进代码，保持测试通过
重复：      下一个功能/场景
```

## TDD 最佳实践

**应该：**
- ✅ 先写测试，在任何实现之前
- ✅ 实现前运行测试验证失败
- ✅ 编写最小代码让测试通过
- ✅ 测试绿色后才重构
- ✅ 添加边缘情况和错误场景
- ✅ 目标 80%+ 覆盖率（关键代码 100%）

**不应该：**
- ❌ 测试前编写实现
- ❌ 每次更改后跳过运行测试
- ❌ 一次写太多代码
- ❌ 忽略失败的测试
- ❌ 测试实现细节（测试行为）
- ❌ 模拟一切（优先集成测试）

## 必须包含的测试类型

**单元测试**（函数级）：
- 顺利路径场景
- 边缘情况（空、null、最大值）
- 错误条件
- 边界值

**集成测试**（组件级）：
- API 端点
- 数据库操作
- 外部服务调用
- 带 hooks 的 React 组件

**E2E 测试**（使用 `/ucc-e2e` 命令）：
- 关键用户流程
- 多步骤过程
- 全栈集成

## 覆盖率要求

- **所有代码最低 80%**
- **以下情况需要 100%：**
  - 财务计算
  - 认证逻辑
  - 安全关键代码
  - 核心业务逻辑

## 重要说明

**强制**：测试必须在实现之前编写。TDD 循环是：

1. **红** - 编写失败的测试
2. **绿** - 实现使其通过
3. **重构** - 改进代码

永不跳过红阶段。永不先写代码再写测试。

## 与其他命令集成

- 先用 `/ucc-plan` 了解要构建什么
- 用 `/ucc-tdd` 带测试实现
- 如遇构建错误用 `/ucc-build-fix`
- 实现完成后用 `/ucc-code-review` 审查
- 用 `/ucc-test-coverage` 验证覆盖率
