# 开发工作流规范

> 此文件定义功能开发、代码审查、发布的标准流程，适用于个人和团队协作。

## 功能开发流程

### 1. 需求分析阶段

**输入：** 用户需求 / Issue / 功能描述

**输出：** 明确的需求文档

**步骤：**
1. 理解需求背景和目标
2. 识别功能边界和约束
3. 确定技术方案选型
4. 评估影响范围

**检查清单：**
- [ ] 需求目标清晰
- [ ] 技术方案可行
- [ ] 影响范围已识别
- [ ] 无遗留问题

### 2. 规划设计阶段

**输入：** 需求文档

**输出：** 实施计划

**步骤：**
1. 调用 `planner` 代理制定计划
2. 分解为可执行的任务单元
3. 识别任务依赖关系
4. 确定关键路径

**检查清单：**
- [ ] 任务分解合理
- [ ] 依赖关系明确
- [ ] 无遗漏步骤
- [ ] 计划可执行

### 3. 开发实现阶段

**输入：** 实施计划

**输出：** 代码 + 测试

**TDD 工作流：**
```
┌─────────────────────────────────────────┐
│                                         │
│  ┌─────────┐   ┌─────────┐   ┌─────────┐│
│  │  红      │ → │  绿      │ → │  重构   ││
│  │ 写测试   │   │ 最小实现 │   │ 优化代码││
│  └─────────┘   └─────────┘   └─────────┘│
│       ↑                             │   │
│       └─────────────────────────────┘   │
│                 循环                     │
└─────────────────────────────────────────┘
```

**检查清单：**
- [ ] 测试先于实现
- [ ] 测试覆盖率 ≥ 80%
- [ ] 所有测试通过
- [ ] 无硬编码值

### 4. 审查阶段

**输入：** 完成的代码

**输出：** 审查报告 + 修复

**步骤：**
1. 调用 `code-reviewer` 进行代码审查
2. 安全敏感代码调用 `security-reviewer`
3. 修复 Critical 和 High 级别问题
4. 确认所有问题已解决

**检查清单：**
- [ ] 无 Critical 问题
- [ ] 无 High 问题
- [ ] Medium 问题已评估
- [ ] 无安全漏洞

### 5. 合并发布阶段

**输入：** 审查通过的代码

**输出：** 合并的代码

**步骤：**
1. 更新文档（调用 `doc-updater`）
2. 运行完整测试套件
3. 创建 Pull Request
4. 合并到主分支

**检查清单：**
- [ ] 文档已更新
- [ ] 测试全部通过
- [ ] PR 描述完整
- [ ] 已获得审批

## 代码审查流程

### 审查触发条件

| 触发条件 | 审查类型 |
|---------|---------|
| 新代码编写完成 | `code-reviewer` |
| 认证/授权相关 | `security-reviewer` |
| 数据库操作相关 | `code-reviewer` + 数据库审查 |
| API 端点变更 | `code-reviewer` + 安全审查 |
| 依赖变更 | `security-reviewer` |

### 审查标准

**代码质量：**
- 可读性：命名清晰、逻辑直观
- 可维护性：函数短小、模块化
- 可测试性：依赖注入、纯函数
- 性能：无明显瓶颈

**安全性：**
- 无硬编码密钥
- 输入已验证
- 输出已编码
- 权限已检查

**测试覆盖：**
- 单元测试覆盖核心逻辑
- 集成测试覆盖 API 端点
- E2E 测试覆盖关键流程

### 审查结果处理

| 问题级别 | 处理要求 |
|---------|---------|
| Critical | 必须立即修复，阻塞合并 |
| High | 必须修复后才能合并 |
| Medium | 建议修复，可记录为技术债务 |
| Low | 可选修复，记录为改进建议 |

## 发布流程

### 发布前检查

```markdown
## 发布检查清单

### 代码质量
- [ ] 所有测试通过
- [ ] 测试覆盖率 ≥ 80%
- [ ] 无 Critical/High 问题
- [ ] 代码审查完成

### 安全检查
- [ ] 无硬编码密钥
- [ ] 依赖无已知漏洞
- [ ] 安全审查完成

### 文档检查
- [ ] README 已更新
- [ ] API 文档已更新
- [ ] CHANGELOG 已更新

### 运维检查
- [ ] 配置已验证
- [ ] 日志已配置
- [ ] 监控已设置
```

### 版本号规范

遵循语义化版本（SemVer）：

```
MAJOR.MINOR.PATCH

MAJOR: 不兼容的 API 变更
MINOR: 向后兼容的功能新增
PATCH: 向后兼容的问题修复
```

### 回滚准备

每次发布前准备：

1. 记录当前版本号
2. 确认回滚命令
3. 验证回滚流程

```bash
# 回滚命令示例
git revert HEAD
# 或
git checkout <previous-tag>
```

## 问题处理流程

### Bug 修复流程

```
报告 Bug → 复现问题 → 编写失败测试 → 修复代码 → 验证修复 → 关闭 Issue
```

### 紧急修复流程

```
发现问题 → 评估影响 → 快速修复 → 紧急发布 → 根因分析 → 防止复发
```

### 技术债务处理

1. **识别：** 在审查中标记 Medium/Low 问题
2. **记录：** 创建 Issue 记录技术债务
3. **评估：** 定期评估优先级
4. **清理：** 安排专门时间处理
