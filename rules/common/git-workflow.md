# Git 工作流规范

> 此文件定义 Git 提交规范、分支策略和 PR 流程，适用于团队协作。

## 提交信息规范

### 格式

```
<类型>: <描述>

[可选的详细描述]

[可选的脚注]
```

### 类型说明

| 类型 | 说明 | 示例 |
|------|------|------|
| `feat` | 新功能 | `feat: 添加用户登录功能` |
| `fix` | Bug 修复 | `fix: 修复登录验证失败问题` |
| `refactor` | 重构（不改变功能） | `refactor: 重构用户服务` |
| `docs` | 文档更新 | `docs: 更新 API 文档` |
| `test` | 测试相关 | `test: 添加用户服务单元测试` |
| `chore` | 构建/工具/依赖 | `chore: 升级依赖版本` |
| `perf` | 性能优化 | `perf: 优化查询性能` |
| `ci` | CI/CD 相关 | `ci: 添加自动化测试流程` |
| `style` | 代码格式（不影响逻辑） | `style: 格式化代码` |

### 描述规则

- 使用祈使句，首字母小写
- 不以句号结尾
- 限制 50 字符以内
- 清晰说明"做了什么"，而非"怎么做的"

**示例：**

```
feat: 添加用户注册功能

- 实现邮箱验证
- 添加密码强度检查
- 集成验证码服务

Closes #123
```

### 禁止的提交

- ❌ `update code`
- ❌ `fix bug`
- ❌ `WIP`
- ❌ `修改了一些东西`
- ❌ 包含敏感信息的提交

## 分支策略

### 分支命名

| 分支类型 | 命名规则 | 示例 |
|---------|---------|------|
| 主分支 | `main` / `master` | `main` |
| 开发分支 | `develop` | `develop` |
| 功能分支 | `feature/<描述>` | `feature/user-auth` |
| 修复分支 | `fix/<描述>` | `fix/login-validation` |
| 发布分支 | `release/<版本>` | `release/1.2.0` |
| 热修复分支 | `hotfix/<描述>` | `hotfix/security-patch` |

### 分支工作流

```
                    ┌─────────────────────────────────────┐
                    │              main                    │
                    └─────────────────────────────────────┘
                              ↑                    ↑
                              │                    │
                    ┌─────────┴─────────┐  ┌──────┴──────┐
                    │    release/1.0.0   │  │  hotfix/xxx │
                    └────────────────────┘  └─────────────┘
                              ↑
                              │
                    ┌─────────┴─────────┐
                    │      develop       │
                    └────────────────────┘
                              ↑
              ┌───────────────┼───────────────┐
              │               │               │
    ┌─────────┴─────────┐ ┌───┴───┐ ┌─────────┴─────────┐
    │ feature/user-auth │ │ fix/xx │ │ feature/dashboard │
    └───────────────────┘ └───────┘ └───────────────────┘
```

### 分支保护规则

**main 分支：**
- 禁止直接推送
- 必须通过 PR 合并
- 需要至少 1 人审批
- 需要 CI 通过

**develop 分支：**
- 禁止直接推送
- 必须通过 PR 合并
- 需要 CI 通过

## Pull Request 流程

### 创建 PR

1. **确保分支是最新的：**
   ```bash
   git checkout develop
   git pull origin develop
   git checkout feature/user-auth
   git rebase develop
   ```

2. **推送分支：**
   ```bash
   git push origin feature/user-auth -u
   ```

3. **创建 PR：** 使用 GitHub/GitLab 界面

### PR 标题格式

与提交信息格式一致：

```
<类型>: <描述>
```

### PR 描述模板

```markdown
## 变更说明

<!-- 简要描述这个 PR 解决的问题 -->

## 变更类型

- [ ] 新功能 (feat)
- [ ] Bug 修复 (fix)
- [ ] 重构 (refactor)
- [ ] 文档 (docs)
- [ ] 其他: ___

## 测试计划

- [ ] 单元测试已添加/更新
- [ ] 集成测试已添加/更新
- [ ] 手动测试已完成

## 检查清单

- [ ] 代码符合项目规范
- [ ] 测试覆盖率 ≥ 80%
- [ ] 文档已更新
- [ ] 无遗留的 console.log/debugger
- [ ] 无硬编码密钥

## 相关 Issue

Closes #xxx
```

### PR 审查

**审查者职责：**
1. 检查代码质量
2. 验证测试覆盖
3. 确认无安全问题
4. 提出改进建议

**审查意见类型：**

| 前缀 | 含义 |
|------|------|
| `MUST` | 必须修改才能合并 |
| `SHOULD` | 建议修改，但非阻塞 |
| `NIT` | 小问题，可选修改 |
| `QUESTION` | 需要澄清 |
| `PRAISE` | 赞扬好的代码 |

### 合并 PR

**合并前确认：**
- [ ] 所有审查意见已处理
- [ ] CI 通过
- [ ] 无冲突
- [ ] 审批数量达标

**合并方式：**

| 方式 | 适用场景 |
|------|---------|
| Merge Commit | 保留完整历史 |
| Squash Merge | 单一功能，保持历史整洁 |
| Rebase Merge | 线性历史 |

## 常见操作

### 撤销最近一次提交（未推送）

```bash
git reset --soft HEAD~1
```

### 修改最近一次提交信息

```bash
git commit --amend
```

### 交互式变基（整理提交历史）

```bash
git rebase -i HEAD~3
```

### Cherry-pick 特定提交

```bash
git cherry-pick <commit-hash>
```

### 解决合并冲突

```bash
# 1. 查看冲突文件
git status

# 2. 手动解决冲突
# 编辑冲突文件，选择保留的代码

# 3. 标记为已解决
git add <resolved-file>

# 4. 继续合并/变基
git merge --continue  # 或 git rebase --continue
```

## 检查清单

**每次提交前：**
- [ ] 代码已格式化
- [ ] 无调试代码
- [ ] 测试通过
- [ ] 提交信息符合规范

**每次推送前：**
- [ ] 已拉取最新代码
- [ ] 已解决冲突
- [ ] CI 本地运行通过

**每次合并前：**
- [ ] PR 已获审批
- [ ] CI 通过
- [ ] 文档已更新
